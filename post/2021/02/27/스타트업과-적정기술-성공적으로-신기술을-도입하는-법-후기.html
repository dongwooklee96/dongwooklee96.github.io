신기술 도입 원칙

  1. 리스크보다 큰 이점이 있는가?
  2. 많이 사용하는 기술인가? -> 내부 구조를 잘 파악하고, 레퍼런스가 많이 있는지?
  3. 팀원들의 공감을 얻었는가?
  4. 지속 가능한 기술인가? -> 관리가 되고 있는 프로젝트 인지, 오버 엔지니어링 경계 -> 의외로 많은 개발자들이 단순히 호기심으로 신기술 도입을 시도한다.

실패 사례의 공통점

  1. 특정 기술에 과하게 몰입
  2. 기술에 몰입한 나머지, 판단력이 흐려진 상태

단점 찾기 / 모르는거 알아보기

  1. 기술 소개 페이지는 장점만 나온다.
  2. 다른 기술과 비교는 필수이다.
  3. 여러개의 단점보다 하나의 단점이 큰 문제를 일으킨다.

회사와 신기술

  1. 지금 사용하는 기술은 언젠가 레거시가 된다.
  2. 신기술 도입은 회사 브랜딩에 도움이 되지만, 개발자를 뽑기 힘들고, 많은 시행착오가 필요하다.
  3. 생각 보다 신기술이 모든 문제를 해결해주지 않는다.
  4. 학습을 하는데 많은 시간이 걸린다.

도입하기 전에 할것

  1. 상대방에게 도입하려는 신기술에 대해서 이야기한다.
  2. 이야기하기 불편한 사람도 설득할 수 있어야 한다.
  3. 진짜 필요한지 생각을 많이 해야하고, 유행에 따라서 도입하는것은 아닌지 생각을 해봐야한다.
  4. 도입하기 전에 사내 내부적으로 스터디를 하고, 사이드 프로젝트를 진행해본다. (중요) -> 프로젝트에 적용하기 전에 무조건 사이드 프로젝트를 해야한다.
  5. 성공적으로 도입했다면, 사내 홍보 및 서서히 부담없는 프로젝트에서 사용을 해본다.

기술보다 중요한 것은 팀원들의 공감이다

  • 새로운 기술을 도입하고, 테스트해야하는 여유가 있는지 확인을 해야한다. -> 회사의 레거시를 줄이고, 인력분배가 먼저 선행되어야 한다.

  • 관성은 변경하기 힘들다 -> 바꾸기 힘들다면 바꿀 수 있는 부분에 집중하는 것이 좋다.

결론

  • 기술보다는 해결하려는 문제를 해결 할 수 있는 방법을 생각하자.

  • 신기술 도입보다는 기존에 사용하는 기술을 좀 더 깊게 사용해보는것도 좋다.
